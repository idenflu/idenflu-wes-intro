# Nova 제품 스토리텔링 문서

> **이 문서의 목적:** 카피 작성, 세일즈 자료, UI 문구 수정 시 일관된 메시지 기준으로 사용한다. 랜딩페이지·발표자료·제안서 모두 이 문서에서 출발한다.

## 1. Product Overview

### 제품명
**Nova** — by Idenflu

### One-liner (한 줄 포지셔닝)
> 창고 운영의 모든 계층 — 설비 모니터링, 작업 지시, 알림, 보안, 스케줄링 — 을 하나의 플랫폼으로 통합한 WES/WCS 솔루션

### Elevator Pitch
레거시 물류 현장은 설비 모니터링·작업 지시·알림·감사가 각각 다른 시스템에 흩어져 있다. 그 결과, 장애 인지는 늦고 의사결정은 보고서에 의존하며 감사 준비는 며칠씩 걸린다.  
Nova는 이 파편화된 구조를 하나의 플랫폼으로 대체한다. 실시간 설비 모니터링부터 RBAC 기반 보안, 멀티채널 알림, 변경 감사, 작업 스케줄링까지 — 단일 계약, 단일 콘솔.  
결과는 간단하다: 장애는 10ms 안에 감지되고, 운영팀은 화면을 전환하지 않으며, 관리자는 감사 전날 밤을 새우지 않는다.

### Why Now — 지금 이 제품이 필요한 이유

- **설비 밀도 증가:** AGV·AMR·소터·컨베이어가 동시에 운영되는 현장이 표준이 됐다. 설비별 개별 솔루션으로는 통합 가시성이 불가능하다.
- **컴플라이언스 압력 강화:** 변경 이력 추적과 접근 감사 요건이 강화되고 있다. 사후 취합 방식으로는 대응 한계가 명확하다.
- **클라우드 네이티브 전환:** 온프레미스 단일 벤더 종속에서 벗어나, 독립 모듈 단위로 확장 가능한 구조로의 전환 수요가 높아지고 있다.

## 2. 핵심 문제 정의

> 레거시 환경의 고통은 "불편함"이 아니라 **조용한 비용 누수**다. 눈에 보이지 않기 때문에 더 위험하다.

### 6가지 고통 — 현장 언어로

| # | 문제 | 현장에서 일어나는 일 | 조용한 비용 |
|---|---|---|---|
| 1 | **파편화된 시스템** | 장애 감지를 위해 모니터링 툴 → 작업 지시 시스템 → 알림 앱을 번갈아 열어야 한다 | 장애 인지 지연 × 빈도 = 누적 처리량 손실 |
| 2 | **지연된 의사결정** | 어제의 보고서로 오늘의 병목을 판단한다. 실시간 대응이 불가능하다 | 잘못된 판단 1건 = 라인 중단 수십 분 |
| 3 | **보안·감사 공백** | "누가 이 설정을 바꿨나요?"에 답할 수 없다. 감사 준비 = 수일간의 로그 수작업 취합 | 감사 실패 리스크 + 담당자 공수 낭비 |
| 4 | **예측 불가한 운영 비용** | 벤더 A 모니터링 + 벤더 B 알림 + 벤더 C 작업 관리. 계약이 쌓일수록 TCO가 불투명해진다 | 라이선스·커스터마이징·유지보수 중첩 |
| 5 | **취약한 시스템 안정성** | AGV 연동 하나가 끊기면 전체 운영이 멈춘다. SPOF가 눈에 보이지 않는다 | 예기치 못한 다운타임 = 출고 지연 = 고객 클레임 |
| 6 | **높은 확장 장벽** | 새 센터를 열거나 설비를 추가할 때마다 IT 재개발이 필요하다. 비즈니스 성장을 IT가 따라가지 못한다 | 확장 프로젝트마다 수개월·수천만 원 |

## 3. 해결 내러티브

### Before Nova / After Nova

| 상황 | Before Nova | After Nova |
|---|---|---|
| 설비 장애 발생 | 담당자가 3개 시스템을 확인 → 수동 전화·메시지 → 재배분 지시 | 10ms 이내 감지 → 멀티채널 자동 알림 → 잔여 설비에 자동 재배분 |
| 감사 준비 | 수일간 시스템별 로그 수작업 취합 → 엑셀 정리 | 기간·대상 조건 설정 → 감사 이력 즉시 내보내기 |
| 신규 설비 추가 | IT 재개발 프로젝트 착수 (수개월) | 독립 모듈로 설비 BC 추가 → 기존 플랫폼과 즉시 연동 |
| 피크 대응 | 물동량 급증 인지 지연 → 수동 재배분 지시 | 임계치 초과 자동 감지 → 유휴 설비 즉시 재배분 |
| 비용 관리 | 벤더별 청구서 취합 후 분기별 검토 | 단일 플랫폼 단일 계약 → 예측 가능한 비용 구조 |

### 핵심 메시지 3가지 (Proof Points)

#### 1. 통합 — "하나의 플랫폼, 완전한 제어"
현장 운영에 필요한 6개 기능 모듈(IAM, Audit, Notification, Realtime, Configuration, Scheduling)이 하나의 플랫폼 안에 있다. 화면 전환 없이 설비 모니터링부터 작업 지시, 알림 확인, 감사 이력까지 단일 콘솔에서 처리한다.

**증명:** 6개 통합 모듈 × 단일 콘솔 × 단일 계약

#### 2. 실시간 — "10ms 응답, 데이터가 운영을 이끈다"
Snapshot-first Delta push 구조로 대시보드와 운영 화면을 지속 동기화한다. 보고서가 아닌 이벤트 스트림이 운영을 이끈다. 장애는 발생 즉시, 알림은 자동으로.

**증명:** 10ms 이하 이벤트 응답 × WebSocket 실시간 스트림 × 멀티채널 자동 알림

#### 3. 확장 — "새 설비를 더해도 재개발 없이"
모듈러 아키텍처로 각 설비 BC(Bounded Context)가 독립적으로 진화한다. 새 설비·새 센터 추가 시 기존 플랫폼을 건드리지 않고 독립 모듈만 연결한다. 비즈니스 성장 속도를 IT가 따라간다.

**증명:** 독립 설비 BC × 기존 운영 무중단 × 수평 확장 가능 아키텍처

## 4. 포지셔닝 맵

### 경쟁 구도

```
               낮은 통합도
                    │
  설비별 전용 툴     │     개별 WMS 솔루션
  (모니터링 전용)    │     (작업 관리만)
                    │
낮은 실시간성 ───────┼─────── 높은 실시간성
                    │
  레거시 WCS        │     ★ Nova
  (온프레미스 단독)  │     (통합 + 실시간 + 확장)
                    │
               높은 통합도
```

### Nova가 차별화되는 3가지 축

| 축 | 기존 솔루션 | Nova |
|---|---|---|
| **통합 범위** | 기능별 개별 도입 | 모니터링·작업·알림·보안·스케줄링 단일 플랫폼 |
| **응답성** | 보고서 기반 (분~시간 단위) | 실시간 이벤트 스트림 (10ms 수준) |
| **확장 방식** | 재개발 필요 (수개월·고비용) | 독립 모듈 추가 (운영 중단 없이) |

## 5. 페르소나별 메시지

### 페르소나 1 — 현장 운영팀

**역할:** 물류센터 현장 운영, 설비 상태 모니터링, 태스크 배분

**핵심 고통:**
- 설비 장애 감지를 위해 여러 화면을 번갈아 확인한다
- 장애 발생 시 직접 전화·메시지로 재배분 지시를 내려야 한다
- 피크 타임에는 눈이 모자란다

**Nova의 답변:**
> "모든 설비 상태와 태스크 현황이 하나의 화면에 있습니다. 장애가 나면 Nova가 먼저 알고, 자동으로 재배분합니다."

**증명 포인트:**
- 2.5D 아이소메트릭 콘솔 — 창고 전체를 한 화면에
- AGV 장애 → 10ms 감지 → 자동 재배분 시나리오
- 심각도별 알림 패널 — 중요한 것만 놓치지 않도록

---

### 페르소나 2 — IT · 시스템 관리자

**역할:** 플랫폼 구축·운영, 보안 정책 관리, 설비 연동

**핵심 고통:**
- 새 설비 추가마다 기존 시스템에 영향이 간다
- 보안 감사 시 접근·변경 이력을 어디서 꺼내야 할지 모른다
- 여러 벤더를 관리하면서 장애 책임 소재가 불분명하다

**Nova의 답변:**
> "각 설비는 독립 모듈로 연결됩니다. 기존 운영을 멈추지 않고 추가하고, RBAC·감사 로그는 플랫폼이 기본 제공합니다."

**증명 포인트:**
- 독립 설비 BC — 기존 플랫폼 영향 없이 연동
- RBAC 기반 접근 제어 + 변경 이력 자동 기록
- 모듈러 아키텍처 — 단일 벤더 종속 해소

---

### 페르소나 3 — 물류 관리자 (의사결정자)

**역할:** 운영 비용 최적화, 리스크 관리, 경영진 보고

**핵심 고통:**
- 여러 솔루션의 라이선스·유지보수 비용이 쌓여 TCO가 불투명하다
- 감사가 갑자기 시작되면 준비하는 데 며칠이 걸린다
- 실시간 운영 데이터가 없어 보고서로만 의사결정을 해야 한다

**Nova의 답변:**
> "단일 플랫폼, 단일 계약. TCO가 예측 가능해집니다. 감사 이력은 즉시 내보낼 수 있고, 실시간 KPI로 데이터 기반 의사결정이 가능합니다."

**증명 포인트:**
- 벤더 통합 → 라이선스·커스터마이징 비용 단순화
- 감사 이력 즉시 내보내기 (기간·대상 조건 설정)
- SLA 99.9% 고가용성 보장

## 6. 산업별 포인트

### 3PL · 물류센터

**핵심 맥락:** 여러 센터·여러 고객사·여러 설비를 동시에 운영한다. 물동량 변동이 크고 피크 대응이 반복된다.

**Nova가 특히 유효한 이유:**
- 센터별 운영 기준과 알림 체계를 분리해 관리
- 피크 타임 태스크 자동 재배분으로 처리량 보호
- 멀티센터 설비 상태를 단일 콘솔에서 통합 관제

**핵심 메시지:** "센터가 늘어도 콘솔은 하나"

---

### 제조 · 공장 물류

**핵심 맥락:** 생산 계획이 곧 물류 작업 지시와 연결된다. AGV·컨베이어·소터가 밀집 운영된다.

**Nova가 특히 유효한 이유:**
- 생산 계획 변경 → 물류 작업 지시 즉시 반영
- 다양한 설비(AGV, 컨베이어, 소터)를 독립 모듈로 통합 연동
- 설비 상태와 작업 지시를 같은 기준·같은 화면에서 관리

**핵심 메시지:** "설비가 달라도, 기준은 하나"

---

### 이커머스 · 풀필먼트

**핵심 맥락:** 오더 수요가 예측하기 어렵고, 피킹·분류·출고 흐름을 빠르게 조정해야 한다.

**Nova가 특히 유효한 이유:**
- 주문 급증 시 작업 우선순위 즉시 재조정
- 피킹·분류·출고 단계를 실시간으로 추적
- 대량 예외 알림과 재처리를 자동화

**핵심 메시지:** "주문이 폭증해도 운영은 흔들리지 않는다"

## 7. 브랜드 보이스 가이드

### 톤 정의
**신뢰할 수 있는 현장 파트너** — 과장하지 않고, 현장 언어로 말한다. 기술을 설명하되 결과로 증명한다.

| 속성 | 설명 |
|---|---|
| **전문적** | 물류·WES/WCS 도메인 언어를 정확하게 사용한다 |
| **간결** | 한 문장에 하나의 메시지. 수식어를 줄인다 |
| **데이터 기반** | 주장에는 수치나 증거를 붙인다 |
| **현장 중심** | "기술"이 아닌 "운영"의 언어로 말한다 |

### 피해야 할 표현

| 피해야 할 표현 | 이유 | 대안 |
|---|---|---|
| "혁신적인", "차세대" | 근거 없는 형용사 | 구체적 기능·수치로 대체 |
| "~할 수 있습니다" (가능성) | 약한 표현 | "~합니다" (사실 진술) |
| "쉽게", "간편하게" | 주관적·검증 불가 | 구체적 단계 수나 시간으로 |
| "모든" (all, every) | 과장 | 실제 커버하는 범위로 한정 |
| "업계 최고", "No.1" | 근거 없이 사용 불가 | 증명 가능한 사실로 |

### 선호 표현 패턴

- **문제 → 원인 → 결과** 순서: "파편화된 시스템은 가시성을 없애고, 대응을 늦춥니다"
- **Before/After 대비**: "보고서 기반 → 실시간 이벤트 스트림"
- **숫자 고정**: "10ms", "99.9%", "6개 모듈" — 수치는 항상 동일하게
- **행동 동사**: "감지합니다", "재배분합니다", "추적합니다" — 수동태 지양

## 9. 메시지 개선 백로그
