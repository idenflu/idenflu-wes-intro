# NOVA 영업자료 문서 초안

> 목적: NOVA 소개자료와 영업 발표자료를 만들기 위한 기준 문서.
> 참고자료의 메시지 구조를 반영하되, NOVA의 강점인 플랫폼 표준화, 실시간 운영, 모듈 확장성을 중심으로 재구성한다.

---

## 0. 참고자료에서 가져올 관점

### NearGo 참고 포인트

참고 URL: [NearGo](https://www.nearsolution.co.kr/neargo)

- 첫 화면에서 "설비 투자 부담 없이", "오늘 바로"처럼 도입 장벽을 낮추는 메시지를 전면에 둔다.
- 기존 랙, 카트, 작업대, ERP, WMS를 활용한다는 식으로 고객의 교체 부담을 줄인다.
- 고객의 고민을 질문형으로 제시한다.
- "작게 시작해서 데이터로 증명하고 확장"하는 흐름이 영업 메시지에 적합하다.

### CJ올리브네트웍스 참고 포인트

참고 URL: [CJ올리브네트웍스 스마트 물류](https://www.cjolivenetworks.co.kr/business/ai_logistics)

- 스마트 물류센터 구축을 컨설팅, 설계, 시스템 구축, 운영 유지보수까지 전체 과정으로 설명한다.
- 중대형 물류센터, E-Commerce, 제조물류, 제약/바이오, F&B, 화장품처럼 적용 산업을 명확히 제시한다.
- 물류IT의 가치를 표준화, 간소화, 동기화, 가시화로 설명한다.
- 자동화 설비와 시스템 운영을 함께 다루며, 구축 이후 유지보수와 장애 대응까지 영업 범위에 포함한다.

### NOVA에 적용할 방향

NOVA는 NearGo처럼 "작게 시작할 수 있음"을 말하되, 단순 경량 솔루션이 아니라 WES/WCS 운영 표준 플랫폼이라는 점을 강조한다.
CJ올리브네트웍스처럼 전체 구축 관점을 가져오되, 대형 SI 메시지보다 모듈 단위 도입과 플랫폼 재사용성을 더 앞에 둔다.

## 1. 영업용 한 줄 정의

> NOVA는 기존 물류 설비와 시스템을 살리면서, 작업 실행·설비 제어·실시간 모니터링·권한·감사·알림을 하나의 표준 플랫폼으로 통합하는 WES/WCS 솔루션입니다.

## 2. 핵심 포지셔닝

### 메인 메시지

> 설비를 전부 바꾸지 않아도, 물류 운영 방식은 바꿀 수 있습니다.

NOVA는 기존 WMS, ERP, 설비, 작업 프로세스를 무시하고 새로 덮는 시스템이 아니다. 현장의 현재 구조를 기준으로 필요한 영역부터 연결하고, 공통 기능은 플랫폼으로 표준화하며, 설비별 차이는 Agent 모듈로 분리한다.

### 보조 메시지

- 처음부터 전체 자동화를 강요하지 않는다.
- 병목 구간부터 적용하고, 운영 데이터를 기준으로 확장한다.
- 공통 기능을 매 프로젝트마다 다시 만들지 않는다.
- 설비 연동, 작업 실행, 알림, 감사, 권한을 같은 운영 구조 안에 둔다.
- 새 설비와 새 센터가 추가돼도 플랫폼 전체를 다시 만들지 않는다.

## 5. NOVA 솔루션 구성

### Platform

운영 전반에 반복적으로 필요한 공통 기능을 제공한다.

| 모듈 | 영업용 설명 |
| --- | --- |
| `IAM` | 사용자, 역할, 권한을 관리해 운영 접근을 통제한다 |
| `Audit` | 변경 이력과 운영 증적을 남겨 감사 대응을 단순화한다 |
| `Notification` | 장애, 작업, 운영 이벤트를 필요한 채널로 전달한다 |
| `Realtime` | 대시보드와 운영 화면을 실시간 상태로 동기화한다 |
| `Configuration` | 센터별, 환경별 운영 설정을 표준 방식으로 관리한다 |
| `Scheduling` | 반복 실행, 예약 작업, 재시도 흐름을 제어한다 |

### Operation

입고, 출고, 이동, 피킹, 분류 같은 업무 상태와 완료 기준을 소유한다.

### Agent

AGV, AMR, Conveyor, Sorter, DAS, DPS, 외부 WMS/ERP처럼 설비나 시스템별 연결 방식을 흡수한다.

### Orchestration

여러 업무와 설비를 순서대로 연결하는 실행 흐름을 조정한다. 재시도, 타임아웃, 보상, 수동 개입 지점을 관리한다.

## 6. 영업 메시지 3축

### 1. 작게 시작

> 전체 교체가 아니라, 병목 구간부터 연결합니다.

- 기존 시스템과 설비를 기준으로 시작한다.
- 피킹, 분류, 출고, 설비 모니터링 등 특정 구간부터 적용할 수 있다.
- 초기 도입 범위를 작게 잡고 운영 데이터로 확장 여부를 판단한다.

### 2. 하나로 운영

> 작업, 설비, 알림, 감사가 같은 운영 구조 안에서 움직입니다.

- 작업 상태와 설비 상태를 같은 콘솔에서 본다.
- 장애나 지연은 알림으로 전달된다.
- 변경 이력과 조치 이력은 감사 증적으로 남는다.

### 3. 계속 확장

> 새 설비가 늘어도 플랫폼은 다시 만들지 않습니다.

- 설비별 차이는 `Agent` 모듈로 분리한다.
- 공통 기능은 `Platform`에서 재사용한다.
- 여러 설비가 참여하는 흐름은 `Orchestration`으로 조정한다.

## 8. 구매자별 메시지

### 의사결정자

> NOVA는 전체 교체보다 낮은 부담으로 시작하고, 확장 시 재개발 리스크를 줄이는 표준 운영 플랫폼입니다.

- 도입 범위와 투자 규모를 단계적으로 설계할 수 있다.
- 공통 플랫폼을 재사용해 센터 확장과 설비 추가 리스크를 줄인다.
- 실시간 운영 데이터와 감사 이력으로 관리 가시성을 높인다.

### 물류 운영 책임자

> 현장 상태를 늦게 보고받는 구조에서, 실시간으로 보고 바로 조치하는 구조로 바꿉니다.

- 작업 진행, 설비 상태, 알림을 한 화면에서 확인한다.
- 피크 시간의 병목과 예외 상황을 빠르게 식별한다.
- 수동 확인과 전화, 메신저 중심 대응을 줄인다.

### IT/시스템 관리자

> 업무, 설비, 공통 기능의 책임을 분리해 유지보수 가능한 구조를 만듭니다.

- `Contracts` 기반으로 BC 간 연결을 관리한다.
- 인증, 권한, 감사, 설정, 스케줄링을 공통 플랫폼으로 표준화한다.
- 설비나 외부 시스템 변화가 전체 플랫폼으로 번지지 않게 한다.

### 설비/자동화 담당자

> 설비별 차이는 인정하고, 운영팀이 보는 기준은 하나로 맞춥니다.

- 설비별 프로토콜과 장애 처리는 `Agent`가 흡수한다.
- 상위 운영 흐름은 `Orchestration`으로 조정한다.
- 설비 상태와 작업 상태를 연결해 운영 판단에 반영한다.

## 10. 도입 시나리오

### 1단계. 현장 진단

- 현재 WMS, ERP, 설비, 작업 프로세스를 확인한다.
- 병목 구간과 수작업 대응 지점을 찾는다.
- 우선 적용할 작업 흐름과 연동 대상을 좁힌다.

### 2단계. 파일럿 적용

- 특정 구역, 특정 작업, 특정 설비군부터 연결한다.
- 작업 상태, 설비 상태, 알림, 이력 데이터를 수집한다.
- 운영팀이 실제로 보는 화면과 대응 절차를 검증한다.

### 3단계. 플랫폼 확장

- `Platform` 공통 기능을 기준으로 권한, 감사, 알림, 스케줄링 범위를 넓힌다.
- 추가 설비는 `Agent` 모듈로 연결한다.
- 여러 업무와 설비를 묶는 흐름은 `Orchestration`으로 확장한다.

### 4단계. 운영 안정화

- 장애 대응, 재시도, 수동 개입 기준을 운영 절차로 고정한다.
- 감사 조회와 운영 리포트 기준을 정리한다.
- 다음 센터 또는 추가 설비 도입 기준을 만든다.

## 11. 바로 쓸 수 있는 영업 문안

### 10초 버전

NOVA는 기존 설비와 시스템을 살리면서 WES/WCS 운영을 표준화된 플랫폼으로 바꾸는 물류 운영 솔루션입니다.

### 30초 버전

NOVA는 물류센터의 작업 실행, 설비 제어, 실시간 모니터링, 권한, 감사, 알림을 하나의 플랫폼 구조로 통합합니다. 전체 교체가 아니라 병목 구간부터 시작하고, 공통 기능은 재사용 가능한 Platform으로, 설비별 차이는 Agent로 분리해 확장 부담을 줄입니다.

### 60초 버전

많은 물류센터가 자동화를 망설이는 이유는 효과가 없어서가 아니라, 초기 투자와 구축 기간, 기존 설비 교체 부담이 크기 때문입니다. NOVA는 이 문제를 전체 교체 방식이 아니라 단계적 플랫폼 도입 방식으로 접근합니다. 기존 WMS, ERP, 설비를 기준으로 필요한 구간부터 연결하고, 권한, 감사, 알림, 실시간 데이터, 스케줄링 같은 공통 기능은 Platform으로 표준화합니다. 이후 새 설비나 새 업무가 추가되면 독립 모듈을 붙이는 방식으로 확장할 수 있습니다.
